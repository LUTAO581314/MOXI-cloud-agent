# 安全边界

MOXI Server Agent 的最大风险是：如果主控或 Agent 被滥用，可能变成大规模服务器控制入口。因此第一版就要按商用安全边界设计。

## 基本原则

- 不保存 root 密码。
- 不在 GitHub 保存密钥、token、证书、数据库密码。
- 不允许任意命令执行作为常规功能。
- 每台服务器单独凭证，支持单独吊销。
- 所有任务必须可审计、可追踪、可回滚。
- 默认最小权限运行，只有必要动作才临时提权。

## Agent 权限

Agent 不应长期以 root 运行全部逻辑。推荐拆分：

- 常驻进程：低权限用户运行。
- 特权动作：通过受控 helper 或 sudoers 白名单执行。

允许的特权动作示例：

- 重启指定 systemd 服务
- reload Nginx
- 执行指定目录下的 docker compose
- 读取指定日志目录
- 清理指定缓存目录

禁止的默认动作：

- 执行任意 shell
- 读取任意系统文件
- 上传本机私钥
- 修改 SSH 授权密钥
- 关闭防火墙

## 通信安全

推荐：

- HTTPS / WSS 加密传输。
- Agent 主动连接主控。
- 请求签名或 mTLS。
- 短期访问 token + 刷新机制。
- 心跳异常时自动降权。

## 审计日志

每个任务记录：

- 操作人
- 服务器节点
- 动作类型
- 参数摘要
- 开始时间
- 结束时间
- 执行结果
- 错误原因

敏感字段必须脱敏，例如 API Key、数据库密码、SSH 私钥、支付密钥。

## 多租户边界

如果未来做托管服务，客户之间必须隔离：

- 一个客户一个 Compose 项目或独立容器组。
- 一个客户一个数据库账号。
- 一个客户一个上传目录。
- 不给客户 root / SSH 权限。
- 客户只能访问自己的应用后台。

## 合规提醒

建议优先做“托管部署服务”，不要一开始做裸服务器出租。裸服务器出租涉及更高滥用风险和 IDC / 增值电信业务合规问题。
