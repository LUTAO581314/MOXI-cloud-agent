# 商用平台实现路线

## Phase 0: 当前基础

已完成：

- 静态前端雏形。
- Django 后端骨架。
- PostgreSQL/Redis/Celery Docker Compose。
- 核心助理模型。
- Django Admin 初版。
- `/portal-api/health/`。

## Phase 1: 可运营 MVP

目标：让平台从静态展示变成能注册、申请、审批、查看状态的真实系统。

任务：

1. 用户认证
   - 注册
   - 登录
   - 登出
   - 当前用户 `/portal-api/me/`

2. 助理配置 API
   - 创建助理草稿
   - 更新模型、场景、目标、语气
   - 更新工具权限
   - 查询我的助理

3. 开通审批
   - 提交开通申请
   - 管理员审批动作
   - 审批后创建 workspace 路径
   - 记录系统日志

4. 前端接入真实 API
   - 配置页提交到后端
   - 控制台读取真实助理
   - 状态页读取真实状态
   - 未登录时引导登录

5. 部署
   - 服务器 `/srv/bairui-chat/backend`
   - Caddy 接 `/portal-api/*` 和 `/admin/*`
   - PostgreSQL 持久化 volume
   - 备份策略初版

验收：

- 新用户能创建助理并提交申请。
- 管理员能在后台批准。
- 用户刷新页面后仍能看到真实助理状态。

## Phase 2: 真实助理运行

目标：让助理能执行最小真实任务，并把结果回写平台。

任务：

1. RuntimeBinding
   - 每个 Assistant 绑定 runtime 类型和 endpoint。
   - 支持 Hermes 内部调用。

2. 任务系统
   - 创建任务
   - Worker 投递任务
   - 任务状态轮询
   - 日志流回写

3. 文件与记忆
   - 本地 workspace 文件索引
   - MemoryItem 管理
   - 用户删除记忆

验收：

- 用户能给助理提交一个文本任务。
- Worker 能执行或代理给 Hermes。
- 任务状态和日志能在控制台看到。

## Phase 3: 商业化

目标：支持套餐、配额、内测资格和基础运营。

任务：

1. Plan / Subscription
   - 套餐权益
   - 当前订阅
   - 试用和内测资格

2. UsageMeter
   - 任务次数
   - 文件空间
   - 工具调用
   - 高级模型额度

3. 邀请码
   - InvitationCode
   - 邀请码兑换
   - 内测名额管理

4. 管理报表
   - 活跃用户
   - 活跃助理
   - 任务量
   - 错误率

验收：

- 管理员能给用户分配套餐。
- 用户能看到自己的权益和用量。
- 超限能力被后端拒绝或要求升级。

## Phase 4: 安全强化

目标：让平台具备商业上线前的安全底线。

任务：

1. 管理员 2FA 或更强保护。
2. 审计日志覆盖审批、权限、登录、删除。
3. 文件上传安全策略。
4. 高权限工具隔离策略。
5. 备份和恢复演练。
6. 隐私删除流程。

验收：

- 关键操作可追溯。
- 普通用户无法访问其他用户资源。
- Hermes 和密钥不暴露给前端。

## Phase 5: 产品体验升级

目标：从可用变成好用。

任务：

1. 独立 React/Vue 前端或继续增强静态 SPA。
2. 助理聊天界面。
3. 文件管理器。
4. 记忆编辑器。
5. 实时日志。
6. 移动端控制台优化。

验收：

- 用户无需理解后端概念即可完成配置和使用。
- 管理员可以日常运营平台。
